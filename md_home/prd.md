## **3\. 背景簡介（目標 / 背景 / 價值）**

### **3.1 產品背景**

都市上班族在平日下班後，常有「今晚想做點什麼」的模糊需求，但會遇到以下問題：

- 選項太多，難以快速決定
- 活動資訊分散在不同平台，查找成本高
- 下班後精力有限，容易出現決策疲勞
- 即使有意願讓生活更充實，最後仍因拖延或猶豫而什麼都沒做

結果是：

- 晚上時間被動流失
- 生活型態趨於單一
- 對日常的主動感與滿足感下降

### **3.2 產品目標**

驗證是否能透過「個人化推薦 \+ 低決策成本設計」，幫助使用者：

- 快速輸入當下狀態
- 在短時間內得到適合今晚的活動選擇
- 降低比較與查詢負擔
- 更容易從「想做點什麼」走到「真的做了什麼」

### **3.3 使用價值**

對使用者的價值：

- 幫使用者節省思考成本
- 在疲累狀態下仍能獲得可行選項
- 增加下班後生活的變化與主動性
- 讓「今晚安排」從高摩擦變成低摩擦

對產品驗證的價值：

- 驗證「狀態輸入 → 推薦活動 → 願意採取行動」是否成立
- 驗證使用者是否認為 3 個推薦已足夠且有幫助
- 驗證 mobile first 情境下是否能快速完成核心任務

### **3.4 要回答的核心問題**

本次 prototype 主要驗證以下問題：

1. 使用者是否願意先輸入當下狀態來換取推薦？
2. 只給 3 個推薦，是否能有效降低決策壓力？
3. 使用者是否能在短時間內理解推薦內容並採取下一步？
4. 推薦卡片中的資訊量是否足夠支撐決策，但不造成負擔？

## **4\. 產品範圍（本次範圍 / 本次不做）**

### **4.1 本次範圍（Prototype Scope）**

本次僅製作前端 demo，展示核心價值流程：

- 使用者進入產品
- 輸入今晚的下班狀態
- 系統根據狀態顯示 3 個今晚推薦
- 使用者查看推薦內容
- 使用者可對推薦進行簡單操作（如重抽、收藏、查看建議原因、點擊行動）

### **4.2 Prototype 要聚焦的驗證點**

- 是否能用極少輸入換得有感推薦
- 是否能讓使用者感覺「不用想太多」
- 是否能促進使用者對某一活動產生行動意圖
- UI 是否適合 mobile first、下班疲累場景

### **4.3 本次不做（Out of Scope）**

以下不在本次 prototype 範圍內：

- 社群功能（揪團、朋友分享、留言互動）
- 使用者辦活動 / 上架活動
- 完整活動平台 / 活動供應商後台
- 真實個人化推薦演算法
- 帳號系統 / 註冊登入
- 真實地理定位與即時距離計算
- 串接外部活動平台或地圖 API
- 付款、預約、票券功能
- 推播通知
- 完整資料分析後台
- 長期偏好學習機制
- 多城市、多語系支援

### **4.4 Prototype 與正式產品範圍區分**

#### **Prototype 範圍**

- 固定 mock data
- 規則式或假資料驅動推薦
- 單一晚間使用情境
- 單人使用流程
- 前端互動展示為主
- 驗證「好不好選」而非「推得多準」

#### **正式產品未來可能範圍（本次不做）**

- 根據歷史偏好、位置、預算、天氣、時間動態推薦
- 真實活動資料源整合
- 收藏與行事曆整合
- 與朋友一起決定今晚活動
- 通勤中與回家後不同情境推薦模式
- 使用者長期興趣成長與探索設計

## **5\. 使用者角色**

### **5.1 目標使用者**

25–35 歲都市白領

### **5.2 角色描述**

- 剛搬到城市
- 生活圈有限
- 平日下班後常想安排點事情
- 但疲累、資訊分散、選擇過多，導致不易做決定
- 希望生活更有變化，不想每天都被動滑手機或發呆結束晚上

### **5.3 角色特徵**

- 使用手機為主
- 使用時間碎片化：通勤途中、晚餐後、回家躺下時
- 決策耐心低
- 偏好清楚、快速、低摩擦介面
- 需要「被整理好的選項」，而非大量清單

### **5.4 主要需求**

- 快速知道今晚能做什麼
- 不想自己開太多 App 比較
- 希望推薦符合當下狀態，而不是抽象的興趣分類
- 希望推薦看起來可執行、門檻不高

### **5.5 主要痛點**

- 不知道從哪開始找
- 覺得麻煩、累、懶得研究
- 太多選擇反而更不想選
- 即使想改變，也缺少低成本的起點

## **6\. 核心功能清單**

### **6.1 核心功能總覽**

1. 下班狀態輸入
2. 今晚推薦生成
3. 推薦卡片瀏覽
4. 推薦理由顯示
5. 重抽推薦
6. 收藏 / 標記有興趣
7. 行動 CTA（例如：今晚去這個 / 先存起來）
8. Empty / Error / 無符合情境的狀態處理

### **6.2 功能說明**

#### **F1. 下班狀態輸入**

讓使用者用最少成本描述今晚狀態，作為推薦依據。

建議輸入維度：

- 今日精力狀態：很累 / 普通 / 還有精神
- 今晚想要的感覺：放鬆 / 有點充實 / 認識新事物 / 動一動
- 預算：不花錢 / 低預算 / 都可以
- 時間：30 分鐘內 / 1–2 小時 / 都可以
- 地點偏好：在家 / 家附近 / 都可以

設計原則：

- 以選項為主，不要求自由輸入
- 避免問太多題
- 單手操作友善
- 支援快速跳過部分欄位

#### **F2. 今晚推薦生成**

根據輸入狀態，顯示 3 個今晚活動推薦。

推薦應具備：

- 明確活動名稱
- 簡短描述
- 為何推薦給你
- 所需時間
- 預算
- 地點形式（在家 / 附近 / 室內外）
- 行動按鈕

#### **F3. 推薦卡片瀏覽**

每張卡片提供足夠決策資訊，但不過度冗長。

卡片應包含：

- 活動標題
- 活動類型標籤
- 時間成本
- 預算級別
- 適合原因
- 簡短行動建議
- 主要 CTA

#### **F4. 推薦理由顯示**

讓使用者理解不是亂推，提升信任感。

推薦理由示例：

- 你今天選擇「很累」，這個活動低腦力、可在家完成
- 你希望今晚有點充實，這個活動可在 1 小時內完成
- 你設定低預算，因此優先推薦不需額外花費的選項

#### **F5. 重抽推薦**

若 3 個推薦都不吸引人，可快速刷新下一組推薦。

需求重點：

- 操作成本低
- 不需重新填整份狀態表單
- 可保留原條件再重抽

#### **F6. 收藏 / 標記有興趣**

讓使用者可先留下覺得不錯的推薦。

Prototype 目的：

- 驗證哪些推薦更容易被保留
- 不需做完整收藏庫頁面，可先用簡化互動展示

#### **F7. 行動 CTA**

每張卡片需有可採取下一步的操作。

CTA 示例：

- 今晚就做這個
- 先存起來
- 再看下一個
- 換一組推薦

#### **F8. 狀態缺失 / 邊界情境處理**

例如：

- 未選任何條件
- 只選一項條件
- 條件過於極端，無明顯匹配
- 重抽到重複活動
- 卡片內容過長

## **7\. 功能流程**

### **7.1 主流程**

1. 使用者進入首頁
2. 看到產品價值主張與開始按鈕
3. 進入「今晚狀態輸入」頁
4. 選擇幾個當下條件
5. 點擊「看今晚推薦」
6. 進入推薦結果頁
7. 查看 3 個推薦卡片
8. 對其中一個活動：
   - 點擊「今晚就做這個」
   - 或點擊「收藏」
   - 或點擊「換一組」
9. 完成一次核心驗證流程

### **7.2 簡化流程圖（文字版）**

首頁  
→ 輸入今晚狀態  
→ 取得 3 個推薦  
→ 查看推薦理由  
→ 選擇：

- 今晚就做這個
- 收藏
- 換一組推薦
- 返回調整條件

### **7.3 次流程 A：重抽流程**

推薦頁  
→ 點擊「換一組推薦」  
→ 保留原條件  
→ 顯示新 3 組推薦  
→ 若資料不足則顯示部分重複，但需提示

### **7.4 次流程 B：調整條件流程**

推薦頁  
→ 點擊「調整條件」  
→ 返回狀態輸入頁  
→ 保留原本選項  
→ 修改後重新推薦

### **7.5 次流程 C：收藏流程**

推薦頁  
→ 點擊卡片上的「收藏」  
→ 顯示成功回饋  
→ 卡片狀態改為已收藏

## **9\. 頁面設計需求（wireframe level）**

---

### **9.1 頁面一：首頁**

#### **頁面目的**

快速傳達產品價值，讓使用者願意開始輸入狀態。

#### **主要內容**

- 主標題：今晚想做點什麼？
- 副標：用 30 秒找到適合今晚的 3 個活動
- 說明文字：不用查一堆資訊，也不用想太多
- 主要按鈕：開始推薦
- 輔助文案：適合下班後、通勤途中、回家後快速使用

#### **UI 重點**

- 首屏就要講清楚價值
- 文案簡短，不用教學式長文
- CTA 明確、單一主行動

#### **驗收要點**

- 使用者 3 秒內可理解這是做什麼的
- 首頁不應出現過多次要入口
- CTA 在首屏可見

---

### **9.2 頁面二：今晚狀態輸入頁**

#### **頁面目的**

用最少選項收集今晚條件，降低填寫負擔。

#### **欄位區塊**

1. 你現在的狀態？
   - 很累
   - 普通
   - 還有精神
2. 今晚想要什麼感覺？
   - 放鬆一下
   - 有點充實
   - 動一動
   - 嘗試新東西
3. 預算
   - 不花錢
   - 300 元內
   - 都可以
4. 時間
   - 30 分鐘內
   - 1 小時左右
   - 1–2 小時
   - 都可以
5. 地點偏好
   - 在家
   - 家附近
   - 都可以

#### **操作元件**

- 選項按鈕 / chips
- 主要按鈕：看今晚推薦
- 次要操作：略過部分條件

#### **UI/UX 原則**

- 單欄垂直排版
- 選項按鈕大、好點擊
- 可快速掃描
- 問題數量控制在 5 題內
- 預設可不全填

#### **驗收要點**

- 任一選項點擊後有明確 selected state
- 未全填仍可送出
- 送出後進入推薦頁
- 表單在手機單手操作合理

---

### **9.3 頁面三：推薦結果頁**

#### **頁面目的**

顯示 3 個今晚推薦，幫助使用者快速決定。

#### **頁面區塊**

1. 頁首摘要
   - 今晚為你挑了 3 個選擇
   - 可顯示簡短條件摘要，如「很累／低預算／在家附近」
2. 推薦卡片列表（3 張）  
   每張卡片包含：
   - 活動名稱
   - 活動類型 tag
   - 一句話介紹
   - 時間
   - 預算
   - 地點形式
   - 為何推薦給你
   - CTA：今晚就做這個
   - 次按鈕：收藏
3. 底部操作
   - 換一組推薦
   - 調整條件

#### **UI/UX 原則**

- 三張卡片資訊密度一致
- 卡片內容以摘要為主，不做長文
- 每張卡片的 CTA 必須明確
- 頁面不要像活動平台列表，而要像「今晚幫你挑好的 3 個」

#### **驗收要點**

- 進頁後立即看到 3 張推薦
- 每張卡片關鍵資訊完整
- 可進行重抽與收藏
- 條件摘要與推薦理由可被理解

---

### **9.4 頁面四：行動確認 / 鼓勵頁（可選）**

#### **頁面目的**

讓使用者在選定活動後，感受到一個簡短的完成感。

#### **顯示內容**

- 文案：今晚就從這個開始吧
- 顯示所選活動名稱
- 小提醒：先不用想太多，做 20 分鐘也算開始
- 次要操作：
  - 再看其他推薦
  - 返回首頁

#### **目的**

此頁主要用於 prototype 展示閉環，不是正式產品必要頁。

---

### **9.5 收藏狀態呈現（不一定獨立頁）**

Prototype 可只做：

- 卡片上的收藏 icon / button
- 點擊後變成已收藏
- Toast：已加入有興趣清單

本次不要求：

- 完整收藏列表管理頁
- 收藏排序與分類

## **10\. Mock Data 規劃**

### **10.1 Mock Data 目的**

由於本次僅為前端 demo，需準備足夠的推薦活動假資料，用於：

- 驗證不同狀態組合下的推薦呈現
- 支援重抽功能
- 模擬個人化推薦理由
- 讓介面看起來接近真實使用情境

### **10.2 Mock Data 欄位建議**

每筆活動資料至少包含：

- activity_id
- 活動名稱
- 活動類型
- 簡短描述
- 適合精力狀態
- 適合目標感受
- 預算級別
- 時間長度
- 地點形式
- 推薦理由模板
- CTA 文案
- 封面圖示 / illustration 類型（可選）
- 是否可收藏

### **10.3 活動類型建議**

建議至少涵蓋以下類型，避免推薦過於單一：

- 在家放鬆型
- 輕運動型
- 自我充實型
- 微社交型
- 城市探索型
- 創作型
- 短時間恢復型

### **10.4 Mock Data 數量建議**

Prototype 建議準備 18–30 筆活動資料，確保：

- 不同條件下有足夠變化
- 重抽不會太容易重複
- 可覆蓋不同精力 / 預算 / 時間 / 地點組合

## **11\. 成功指標**

### **11.1 Prototype 階段成功指標**

本次以產品方向驗證為主，不以商業轉換為主。

#### **定性指標**

- 使用者是否覺得流程輕鬆、不麻煩
- 使用者是否認為推薦有貼近今晚狀態
- 使用者是否認為只給 3 個剛剛好
- 使用者是否願意點擊某個推薦的 CTA
- 使用者是否能說出「這比我自己想快很多」

#### **定量指標（Prototype 測試可觀察）**

- 完成一次推薦流程的比例
- 從首頁到推薦頁的完成率
- 單次流程完成時間
- 推薦頁 CTA 點擊率
- 重抽率
- 收藏率
- 返回調整條件率

### **11.2 建議目標值（驗證用假設）**

以下為合理假設，供 usability test 參考：

- 80% 以上使用者能在 1 分鐘內完成一次推薦流程
- 60% 以上使用者對至少 1 個推薦表示「願意考慮」
- 50% 以上使用者認為 3 個推薦已足夠
- 70% 以上使用者能理解推薦理由

## **12\. 驗收標準**

### **12.1 整體驗收**

- 可完整展示「輸入狀態 → 得到 3 個推薦 → 採取操作」流程
- Prototype 為 mobile first
- 不依賴後端與真實 API 即可操作展示
- 主要流程在前端 demo 中可順暢走通

### **12.2 功能驗收**

#### **A. 首頁**

- 顯示產品標題、價值主張、開始按鈕
- 點擊開始按鈕後可進入狀態輸入頁

#### **B. 狀態輸入頁**

- 至少包含精力、期待、預算、時間、地點等條件
- 每題有可點選選項
- 使用者不需填滿所有欄位也可進入下一步
- 已選狀態有清楚視覺回饋
- 點擊「看今晚推薦」後進入推薦頁

#### **C. 推薦結果頁**

- 顯示 3 張推薦卡片
- 每張卡片含活動名稱、描述、時間、預算、地點、推薦理由、CTA
- 可點擊「換一組推薦」
- 可點擊「調整條件」
- 可點擊「收藏」
- 收藏後有成功回饋

#### **D. 重抽邏輯（前端 demo 層級）**

- 點擊換一組後，畫面應更新為另一組推薦
- 若資料不足導致部分重複，需有合理處理，不可呈現明顯故障狀態

#### **E. 完成感頁（若有）**

- 點擊主要 CTA 後可進入簡單完成頁
- 能顯示使用者選中的活動名稱

### **12.3 UI 驗收**

- 手機寬度下版面不破版
- 主 CTA 一致且明確
- 文字可讀性足夠
- 卡片層級分明
- 避免一次顯示過多資訊造成壓力

### **12.4 內容驗收**

- 文案符合下班疲累情境
- 推薦活動內容不脫離都市白領生活情境
- 推薦理由能對應使用者所選條件
- 不出現與當前條件完全衝突的推薦內容

## **13\. Corner Case**

### **13.1 輸入相關**

1. **使用者一個條件都沒選**
   - 處理方式：允許進入推薦頁，顯示通用型今晚推薦
2. **使用者只選一個條件**
   - 處理方式：仍可推薦，並降低推薦精準描述強度
3. **條件組合很極端**
   - 例如：很累 \+ 想動一動 \+ 不花錢 \+ 30 分鐘內 \+ 在家
   - 處理方式：仍給出相對接近的低門檻活動，不顯示「查無結果」

### **13.2 推薦相關**

4. **3 個推薦高度相似**
   - 處理方式：mock data 與前端展示要避免連續出現內容幾乎相同的卡片
5. **重抽後出現重複推薦**
   - 處理方式：可接受少量重複，但需盡量降低；若重複，避免 3 張全重複
6. **推薦內容與條件明顯衝突**
   - 例如使用者選很累，卻推薦高強度社交夜店活動
   - 處理方式：應在 mock data 標記時先避免

### **13.3 操作相關**

7. **使用者連續快速點擊換一組**
   - 處理方式：前端需避免出現閃爍、卡死、狀態錯亂
8. **收藏後再取消收藏**
   - 處理方式：若 prototype 有此設計，需狀態一致；若不支援取消，需明確只做單向收藏展示
9. **返回上一頁後資料消失**
   - 處理方式：返回調整條件時，原本選項應保留

### **13.4 內容呈現相關**

10. **活動名稱過長**

- 處理方式：需有文字截斷或自適應排版

11. **推薦理由過長**

- 處理方式：控制為 1–2 句，避免卡片過高

12. **小螢幕裝置**

- 處理方式：優先保障 CTA、活動名稱、推薦理由可見

## **14\. 備註 / 假設**

### **14.1 已知限制**

- 僅做前端 demo 流程
- 使用 mock data
- mobile first
- 不處理真實活動來源與後端資料邏輯

### **14.2 合理假設**

以下為本 PRD 補充之合理假設，供設計與前端實作依循：

1. **推薦不是 AI 真實生成**
   - 本次以 mock data \+ 前端規則或固定資料組合模擬推薦結果
2. **推薦以「今晚可執行」為原則**
   - 不推薦過度依賴報名、付款、路程規劃的活動
3. **活動資料以低門檻為主**
   - 優先選擇可以快速開始、心理阻力低的活動
4. **使用者主要使用手機操作**
   - 不優先為桌機設計複雜 layout
5. **本次不追求推薦精準率，只驗證決策體驗**
   - 核心是降低選擇負擔，不是建立完整推薦系統
6. **收藏功能僅作互動驗證**
   - 不需建立完整個人收藏管理系統
7. **不做真實定位與天氣情境**
   - 地點與環境以 mock 條件模擬

### **14.3 設計提醒**

- 這不是活動平台首頁，而是「今晚幫你做初步決策」的工具
- 資訊不是越多越好，重點是讓人願意選
- 3 個推薦是刻意設計，不應在 prototype 中擴張成長列表
- 推薦卡片的語氣要像幫使用者減壓，而不是逼他自我提升；別把產品做成下班後第二份工作，這很重要
